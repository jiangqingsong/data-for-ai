## 一、定位与目标

### 1.1 当前画像

| 维度 | 现状 |
|------|------|
| 工作经验 | 10年大数据开发 |
| 行业 | 汽车整车制造 |
| 核心技术 | Spark/Flink/Hive/SQL/Python/数据建模 |
| AI/ML | 有接触但不深（LLM应用数据支撑） |

### 1.2 转型目标

```
当前：大数据开发工程师（10年）
  │
  │  6-12个月
  ▼
中期：AI 数据工程 / DATA for AI 工程师
  │
  │  2-3年
  ▼
远期：AI 应用数据架构师 / ML Data Platform 负责人
```

差异化优势：大数据工程能力（10年）+ 汽车行业经验 + AI 数据工程新技能 = 复合稀缺型人才

### 3.2 项目一：RAG 知识库问答系统

**目标：** 3周内从零搭建一个可用的 RAG 问答系统，理解全链路。

**场景选择：** 初中物理/数学/历史知识库问答（与 ExamAI 后续项目衔接，数据源公开无版权问题）

**技术栈：** Python + LangChain + Chroma + 通义千问 API（或 DeepSeek API）+ Streamlit/Gradio

**里程碑：**

| 周次 | 主题 | 产出 | 预估时间 |
|------|------|------|---------|
| **第1周** | 最简 RAG 跑通 | 100行脚本：PDF→分块→向量化→检索→生成 | 5-8h |
| **第2周** | 检索质量优化 | 分块策略实验 + 检索策略对比 + 评估脚本 | 5-8h |
| **第3周** | 产品化 | Web界面 + 对话历史 + 数据源管理 + 知识库管理 | 8-12h |

#### 第1周详细任务：最简 RAG 跑通

**Day 1（1-2h）：环境搭建**
- [ ] 创建项目目录 `rag-knowledge-qa/`
- [ ] 安装依赖：`pip install langchain chromadb python-dotenv unstructured pypdf`
- [ ] 配置 LLM API Key（通义千问或 DeepSeek）
- [ ] 跑通 LangChain Hello World

**Day 2（1-2h）：准备知识库数据**
- [ ] 搜集初中物理教材/知识点 PDF 5-10 份（公开资源）
- [ ] 使用 LangChain Document Loader 加载 PDF
- [ ] 理解 Document 对象结构

**Day 3（1-2h）：文档分块与向量化**
- [ ] 使用 RecursiveCharacterTextSplitter 分块
- [ ] 实验不同 chunk_size（500/1000/1500）
- [ ] 使用通义千问 Embedding 向量化
- [ ] 存入 Chroma 向量数据库

**Day 4（1-2h）：检索 + 生成**
- [ ] 实现相似度检索
- [ ] 实现 RAG Chain：检索 → 拼 Prompt → LLM 生成
- [ ] 测试几个问题，观察回答质量

**Day 5（1-2h）：整理与总结**
- [ ] 代码整理、加注释
- [ ] 写一篇简短的总结：架构、遇到的问题、下一步优化方向
- [ ] 提交 GitHub

#### 第2周详细任务：检索质量优化

**Day 1（1-2h）：分块策略实验**
- [ ] 对比固定大小分块 vs 按段落分块 vs 按标题分块
- [ ] 记录不同策略的效果差异
- [ ] 选择最优策略

**Day 2（1-2h）：检索策略优化**
- [ ] 实现 MMR（最大边际相关性）检索，增加结果多样性
- [ ] 对比相似度检索 vs MMR 检索效果
- [ ] 加入检索重排序（rerank）

**Day 3（1-2h）：评估体系搭建**
- [ ] 准备 20 个测试问题 + 标准答案
- [ ] 写评估脚本：检索是否包含答案？（召回率）
- [ ] 写评估脚本：生成答案是否准确？（准确率）

**Day 4（1-2h）：迭代优化**
- [ ] 根据评估结果调整参数
- [ ] 对比优化前后的评估指标
- [ ] 记录优化心得

**Day 5（1-2h）：整理总结**
- [ ] 更新 GitHub 代码
- [ ] 写优化总结文档

#### 第3周详细任务：产品化

**Day 1-2（2-4h）：Web 界面**
- [ ] 用 Streamlit 或 Gradio 搭建 Web UI
- [ ] 功能：输入问题 → 显示回答 → 显示参考文档来源
- [ ] 美化界面

**Day 3-4（2-4h）：高级功能**
- [ ] 多轮对话支持（对话历史记忆）
- [ ] 知识库管理（上传文档、删除、重新向量化）
- [ ] 知识库列表（切换不同学科的知识库）

**Day 5（1-2h）：收尾**
- [ ] 完整测试
- [ ] 写 README（项目介绍、架构图、使用说明）
- [ ] 提交 GitHub，打 v1.0 Tag

### 3.3 项目二：ExamAI 改造 — AI 数据工程版

**目标：** 在项目一 RAG 基础上，做一个完整的 AI 数据工程项目。核心从"业务 CRUD"转向"数据 pipeline"。

**与原 ExamAI 设计的差异：**

| 模块 | 原设计 | AI 数据工程版改造 |
|------|--------|-----------------|
| 试卷录入 | 手动上传 + LLM 解析 | → **非结构化数据 pipeline**：PDF/Word→解析→分块→结构化→质量校验 |
| 题目检索 | SQL 关键词搜索 | → **语义检索**：向量相似度搜索 + 混合检索 |
| 组卷推荐 | A面配置 + B面 AI 推荐 | → **RAG 增强组卷**：基于知识点向量检索候选题目 |
| 题库管理 | CRUD | → **数据版本管理**：题目修改历史、数据集快照 |
| 新增模块 | — | → **数据质量看板**：题目完整性、难度分布、知识点覆盖统计 |
| 新增模块 | — | → **检索评估**：组卷结果相关性评估 |

**技术栈：** Django REST + React + LangChain + Chroma/Milvus + Unstructured + Celery + PostgreSQL

**里程碑（待项目一完成后细化）：**

| 阶段 | 内容 | 预估时间 |
|------|------|---------|
| M1 | Django 脚手架 + 数据模型 + 非结构化处理 pipeline | 2周 |
| M2 | 语义检索引擎（向量化 + 混合检索） | 2周 |
| M3 | RAG 增强组卷引擎 | 2周 |
| M4 | 前端 + 数据质量看板 | 2周 |
| M5 | 整合测试 + 文档 + GitHub | 2周 |

## 七、风险与应对

| 风险 | 应对 |
|------|------|
| 工作太累晚上学不动 | 降低单日目标，保证持续性比强度重要 |
| 项目进度落后 | 砍功能保核心，先跑通再完善 |
| 学习内容太多焦虑 | 严格按优先级，🔴必学先搞，🟢可选放掉 |
| 被优化/换工作 | 有项目+有文档+有博客，简历已更新，不慌 |
